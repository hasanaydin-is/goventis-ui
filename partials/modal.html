Acceptance criteria

  • SAML 2.0 ve OAuth2 (Authorization Code + PKCE) ile SSO girişi yapılabilmeli.
  • Tenant başına IdP konfigürasyonu admin panelinden yapılabilmeli.
  • Just-in-time provisioning ile ilk girişte kullanıcı otomatik oluşturulmalı.
  • SSO failover senaryosunda local credential fallback aktif olmalı.
  • Audit log: her giriş denemesi auth.sso.attempt event'i olarak loglanmalı.

Problem

Enterprise müşterilerimizin %62'si onboarding sırasında "kendi IdP'mizle (Okta / Azure AD / Google Workspace) girebilmeliyiz" diyor. Şu an sadece email + magic link destekliyoruz; bu hem güvenlik ekiplerinin, hem de bizim provisioning sürecimizin tıkandığı yer.

Çözüm

Goventis backend'ine OAuth2 (Authorization Code + PKCE) ve SAML 2.0 destekli bir SSO katmanı eklenecek. Tenant-aware bir auth/sso/connector servisi, her tenant için saklanan IdP konfigürasyonunu okuyup ilgili flow'u başlatacak.

Akış

  1. Kullanıcı goventis.app/sso/<tenant> adresine gider.
  2. Connector tenant config'i çözer, ilgili IdP'ye redirect başlatır.
  3. Callback'te ID token doğrulanır, JIT provisioning ile user kaydı açılır/günceller.
  4. Session token issue edilir, dashboard'a redirect.
Not: bu iş tamamlanmadan GOV-127 (audit log persistence) merge edilmemeli — SSO event'leri o pipeline'a yazılacak.

Test stratejisi

  • Unit: connector resolver, token validator (Jest)
  • Integration: 3 sandbox IdP — Okta dev, Azure B2C, Google Workspace
  • QA: AI test suite GOV-142-suite üretildi (12 case), sprint 24 sonuna kadar koşulacak
